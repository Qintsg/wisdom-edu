# 大模型接入说明

## 当前支持的提供方

项目的大模型能力统一由 `backend/ai_services/services/llm_service.py` 管理，当前默认走 **OpenAI 兼容协议**，默认提供方为 `deepseek`，默认模型为 `deepseek-v4-flash`，并显式关闭 reasoning / thinking 输出。

| 提供方 | `LLM_PROVIDER` | 典型模型 | 主要密钥变量 | 默认兼容地址 |
| --- | --- | --- | --- | --- |
| 通义千问 | `qwen` | `qwen3.6-plus`、`qwen3.5-plus`、`qwen-plus`、`qwen3-coder-plus` | `DASHSCOPE_API_KEY` | `https://dashscope.aliyuncs.com/compatible-mode/v1` |
| DeepSeek | `deepseek` | `deepseek-v4-flash`、`deepseek-chat`、`deepseek-reasoner`、`deepseek-coder` | `DEEPSEEK_API_KEY` | `https://api.deepseek.com` |
| 豆包 / ModelArk | `doubao` | `doubao-seed-1.6`、`doubao-seed-1.6-thinking`、`ByteDance-Seed-1.8` | `ARK_API_KEY` / `DOUBAO_API_KEY` | `https://ark.ap-southeast.bytepluses.com/api/v3` |
| 智谱 / GLM | `zhipu` | `glm-5`、`glm-4.7`、`glm-4.6` | `ZAI_API_KEY` / `ZHIPU_API_KEY` | `https://open.bigmodel.cn/api/paas/v4/` |
| Kimi / Moonshot | `kimi` | `kimi-k2.5`、`kimi-k2-thinking`、`moonshot-v1-*` | `MOONSHOT_API_KEY` / `KIMI_API_KEY` | `https://api.moonshot.ai/v1` |
| 自定义兼容网关 | `custom` | 任意兼容模型名 | `CUSTOM_LLM_API_KEY` / `LLM_API_KEY` | 由 `CUSTOM_LLM_BASE_URL` 或 `LLM_BASE_URL` 指定 |

底层客户端仍然是 `langchain_openai.ChatOpenAI`，因此项目不依赖 OpenAI 官方服务本身，只要求目标提供方支持兼容的 `base_url + api_key + model` 调用方式。

## 配置来源与优先级

运行时配置现在同时支持两层来源：

1. `backend/config.ini`：保存非敏感默认值，例如 `provider`、`model`、`api_format`、`timeout`、`GraphRAG` 向量参数。
2. `backend/.env`：保存密钥、环境级覆盖值与部署差异化配置。

解析优先级为：

1. Django `settings` 中已解析出的环境变量；
2. `backend/.env`；
3. `backend/config.ini`；
4. 代码内置默认值。

也就是说：

- `config.ini` 负责“默认怎么跑”；
- `.env` 负责“这个环境具体用谁来跑”。

## 关键配置项

### 通用配置

| 变量 | 说明 |
| --- | --- |
| `LLM_PROVIDER` | 显式指定默认提供方，推荐始终配置 |
| `LLM_MODEL` | 默认模型名 |
| `LLM_API_FORMAT` | 接口格式描述，当前推荐 `openai-compatible` |
| `LLM_REQUEST_TIMEOUT` | 单次请求超时秒数 |
| `LLM_MAX_RETRIES` | LangChain 客户端最大重试次数 |
| `LLM_REASONING_ENABLED` | 是否允许模型 reasoning / thinking 模式，默认 `false` |
| `LLM_REASONING_EFFORT` | reasoning 强度，仅在 `LLM_REASONING_ENABLED=true` 时透传 |
| `LLM_EXTRA_BODY_JSON` | 透传给兼容网关的 JSON 请求体，默认包含 `{"enable_thinking": false}` |
| `LLM_API_KEY` | 统一兼容网关密钥，可作为共享回退 |
| `LLM_BASE_URL` | 统一兼容网关地址，可全局覆盖提供方默认地址 |
| `HTTP_PROXY` / `HTTPS_PROXY` | 标准 HTTP(S) 代理地址，所有兼容网关均可复用 |
| `LLM_HTTP_PROXY` / `LLM_HTTPS_PROXY` | 仅对 LLM 网关生效的专用代理，优先级高于标准代理 |

### 提供方专用配置

| 提供方 | 专用密钥 | 专用 Base URL |
| --- | --- | --- |
| 千问 | `DASHSCOPE_API_KEY` | `QWEN_BASE_URL` |
| DeepSeek | `DEEPSEEK_API_KEY` | `DEEPSEEK_BASE_URL` |
| 豆包 | `ARK_API_KEY` / `DOUBAO_API_KEY` | `DOUBAO_BASE_URL` |
| 智谱 | `ZAI_API_KEY` / `ZHIPU_API_KEY` | `ZHIPU_BASE_URL` |
| Kimi | `MOONSHOT_API_KEY` / `KIMI_API_KEY` | `KIMI_BASE_URL` |
| 自定义兼容网关 | `CUSTOM_LLM_API_KEY` | `CUSTOM_LLM_BASE_URL` |

## 提供方识别逻辑

`LLMService` 现在的识别顺序是：

1. 优先尊重显式 `LLM_PROVIDER`；
2. 若未显式指定，则按 `LLM_MODEL` 前缀回推提供方；
3. 若仍无法判断，但存在 `LLM_API_KEY/CUSTOM_LLM_API_KEY + LLM_BASE_URL/CUSTOM_LLM_BASE_URL`，则回退为 `custom`；
4. 最后保底回退到 `deepseek`。

Base URL 的解析顺序为：

1. `LLM_BASE_URL`（统一兼容网关）
2. 提供方专用 `*_BASE_URL`
3. 代码内置的官方默认地址

API Key 的解析顺序为：

1. 当前提供方专用密钥
2. `LLM_API_KEY`
3. `CUSTOM_LLM_API_KEY`（当提供方为 `custom` 时优先）

代理地址的解析顺序为：

1. `LLM_HTTPS_PROXY` / `LLM_HTTP_PROXY`
2. `HTTPS_PROXY` / `HTTP_PROXY`
3. 不走代理

当模型网关是 `https://...` 时，系统优先使用 HTTPS 代理；若只配置了 HTTP 代理，则会自动回退到 HTTP 代理地址。

## 配置示例

### DeepSeek（默认推荐，非思考模式）

```env
LLM_PROVIDER=deepseek
LLM_MODEL=deepseek-v4-flash
LLM_API_FORMAT=openai-compatible
LLM_REASONING_ENABLED=false
LLM_REASONING_EFFORT=
LLM_EXTRA_BODY_JSON={"enable_thinking":false}
DEEPSEEK_API_KEY=your-deepseek-key
DEEPSEEK_BASE_URL=https://api.deepseek.com
```

默认配置会对 DeepSeek v4、DeepSeek reasoner、Qwen thinking 等容易返回推理片段的模型透传 `enable_thinking=false`。如果确需开启推理模式，需要同时设置 `LLM_REASONING_ENABLED=true`，并按目标网关要求调整 `LLM_REASONING_EFFORT` 与 `LLM_EXTRA_BODY_JSON`。

学习节点的外部资源推荐现在优先走学习资源 MCP：项目内资源由自研内部资源 MCP 从数据库检索，外部资源优先通过 Exa 语义搜索召回，再用 Firecrawl 抓取正文摘要。未配置 `EXA_API_KEY` 或外部服务不可用时，才回退到 DeepSeek / 当前兼容网关的 `enable_search=true` 联网推荐。

### 通过代理访问外部模型网关

```env
LLM_PROVIDER=deepseek
LLM_MODEL=deepseek-chat
DEEPSEEK_API_KEY=your-deepseek-key
HTTPS_PROXY=http://127.0.0.1:7890
```

如果只希望大模型调用走代理，而不影响系统中的其他 HTTP 请求，可以改用：

```env
LLM_HTTP_PROXY=http://127.0.0.1:7890
LLM_HTTPS_PROXY=http://127.0.0.1:7890
```

### 学习资源 MCP（Exa + Firecrawl）

```env
RESOURCE_MCP_ENABLED=True
RESOURCE_MCP_EXA_ENABLED=True
RESOURCE_MCP_FIRECRAWL_ENABLED=True
EXA_API_KEY=your-exa-key
FIRECRAWL_API_KEY=your-firecrawl-key
```

`EXA_API_KEY` 用于外部学习资源语义搜索，`FIRECRAWL_API_KEY` 用于对 Exa 命中页面抓取正文摘要。两者均为空时，学生端资源推荐仍会返回课程内资源，并保留原 LLM 联网推荐兜底。

## 代码内调用链路

业务代码通常不会直接创建模型客户端，而是统一经过：

1. `platform_ai.llm.llm_facade`
2. `ai_services.services.llm_service.LLMService`
3. LangChain agent 或直接聊天模型调用

这条链路现在会共享同一套：

- `provider / model / api_format / timeout / retries`
- `reasoning_enabled / reasoning_effort / extra_body`
- API Key 解析逻辑
- Base URL 回退逻辑
- 代理解析逻辑
- `<think>` 推理片段剥离、JSON 修复与 fallback 机制

## 与 LangChain agent / GraphRAG 的关系

- LangChain agent 现在会沿用同一套 LLM 配置，不需要单独再配一份 agent 专用模型。
- GraphRAG 路由器 `FacadeGraphRAGLLM` 和学生端问答链路仍走课程证据检索，但其底层默认模型来源也已统一到 `settings`。
- 对前端来说不需要新增配置入口；切换提供方仍主要是后端 `.env + config.ini` 的职责。

## 输出与降级策略

`LLMService` 的目标不是“强行调用成功”，而是“尽量返回稳定的结构化结果”。

当前策略为：

1. 优先尝试 LangChain agent；
2. 再回退到底层聊天模型；
3. 若输出不是合法 JSON，则尝试 JSON 修复；
4. 仍失败时，返回业务层预设 fallback。

因此即使：

- 未配置可用密钥；
- 模型服务短暂异常；
- 模型输出混入 Markdown 或解释文字；

页面通常也不会直接崩溃，而是回到可展示的业务默认结果。

## 排障建议

### 1. 明明配了密钥，但仍走 fallback

优先检查：

- `backend/.env` 是否真的被加载；
- `LLM_PROVIDER` 与密钥类型是否匹配；
- `LLM_BASE_URL` 是否误填成了旧网关地址；
- `backend/config.ini` 是否覆盖了你预期之外的默认模型。

### 2. 自定义网关调用失败

优先检查：

- `CUSTOM_LLM_BASE_URL` 是否为兼容接口根地址；
- `LLM_API_FORMAT` 是否仍为当前支持的 `openai-compatible` / `chat-completions`；
- 网关是否要求额外头部或特殊模型名映射。

### 3. WebSocket AI 对话不可用

优先检查：

- 后端是否按 ASGI 链路启动；
- 反向代理是否正确转发 `/ws/`；
- 浏览器控制台是否有 `/ws/student/ai/chat` 连接失败或鉴权错误。
