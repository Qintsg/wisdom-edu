# 服务器部署说明

> 推荐部署形态：`106 公网入口服务器（Nginx + frontend/dist + frps）` + `内网后端服务器（Daphne(ASGI) + PostgreSQL + Neo4j）`。

## 适用场景

本文针对以下实际部署条件编写：

- 一台**无公网地址**的内网后端服务器：承载 Django ASGI、PostgreSQL、Neo4j。
- 一台拥有公网入口与域名 `edu.qintsg.xyz` 的 `106` 服务器：承载 `frps`、Nginx 与 `frontend/dist`。
- `frps` 已将内网后端映射到 `106` 服务器的 `28000` 端口（推荐本机访问地址：`http://127.0.0.1:28000`；外部可见地址通常为 `http://edu.qintsg.xyz:28000`）。
- 当前交付的前端生产包默认使用**同域** `/api`、`/media`、`/static`、`/ws`，浏览器应统一访问：
  - `https://edu.qintsg.xyz/api/`
  - `https://edu.qintsg.xyz/media/`
  - `https://edu.qintsg.xyz/static/`
  - `wss://edu.qintsg.xyz/ws/`

这意味着：`106` 服务器上的 Nginx 不仅要托管 `frontend/dist`，还需要把 `/api`、`/ws`、`/media`、`/static` 反向代理到 `frps` 暴露出来的后端入口，这样可以同时解决**静态部署刷新 404**、**跨域**与 **HTTPS 页面访问 HTTP API 的混合内容**问题。

## 推荐拓扑

本项目后端启用了 `Channels` 和 WebSocket 路由 `/ws/student/ai/chat`，因此生产环境应按 ASGI 服务部署，而不是只按传统 WSGI 部署。

推荐的双机拓扑如下：

```text
浏览器
  │
  ▼
106 公网入口服务器
  - Nginx
  - frontend/dist
  - frps（映射内网 Django 到本机 28000）
  - 统一对外域名 https://edu.qintsg.xyz
  │
  ▼
内网后端服务器
  - Daphne + Django ASGI
  - PostgreSQL
  - Neo4j
  - backend/media
  - backend/staticfiles
```

推荐域名规划：

- 前端站点：`https://edu.qintsg.xyz/`
- frps 上游入口：`http://127.0.0.1:28000/`（如需公网直连排障，也可临时使用 `http://edu.qintsg.xyz:28000/`）
- 浏览器实际访问：
  - 页面：`https://edu.qintsg.xyz/`
  - API：`https://edu.qintsg.xyz/api/`
  - WebSocket：`wss://edu.qintsg.xyz/ws/student/ai/chat`

也就是说，浏览器对外只看到统一的 `edu.qintsg.xyz` 入口；前端静态资源与后端接口都走同域访问，由 `106` 服务器上的 Nginx 转发到 frps 暴露的 `28000` 后端上游。这是当前部署条件下最稳妥的方案。

## 资源分工建议

### 后端服务器（2核4G）

建议承载：

- `1Panel` 面板本体
- `Daphne + Django`
- `PostgreSQL`
- `Neo4j`
- `backend/media/`
- `backend/staticfiles/`
- `backend/runtime_logs/`
- AI / GraphRAG / KT 所需运行时文件

不建议在这台机器上长期运行：

- 前端开发服务器
- 浏览器巡检进程
- 模型训练任务
- 无限制的 Neo4j 大内存配置

### 106 公网入口服务器

建议承载：

- `Nginx`
- `frontend/dist`
- `frps`
- 站点 HTTPS 证书

不建议承载：

- PostgreSQL
- Neo4j
- Django / Daphne
- Playwright 或其他重型自动化进程常驻

## 端口与网络规划

### 建议对公网开放

106 公网入口服务器：

- `80`
- `443`

后端服务器：

- `1Panel` 面板端口（请自定义，不使用默认弱口）

### 建议仅本机或内网开放

后端服务器：

- `8000`（Daphne，仅供本机 Nginx / 1Panel 反代）
- `5432`（PostgreSQL）
- `7687`（Neo4j Bolt）
- `7474`（Neo4j HTTP 管理端口，如启用）

106 公网入口服务器：

- `28000`（frps 暴露的 HTTP 后端入口，推荐仅供本机 Nginx 转发）

## 后端服务器部署（2核4G + 1Panel）

### 1. 安装 1Panel

根据 1Panel 官方文档，在线安装方式如下：

```bash
bash -c "$(curl -sSL https://resource.1panel.pro/quick_start.sh)"
```

安装完成后：

1. 记录面板访问地址、安全入口和管理员账号。
2. 立即修改面板默认端口、入口路径和密码。
3. 用 `1pctl user-info` 再次确认面板访问信息。

### 2. 在后端服务器准备运行目录

```text
/srv/wisdom-edu/
    backend/
```

建议把业务代码与 1Panel 面板数据分开，避免后续迁移和备份互相打架。

### 3. 部署 PostgreSQL 与 Neo4j

可以通过 `1Panel` 的容器 / 应用编排能力部署数据库，也可以手工安装；无论哪种方式，都建议遵循以下原则：

- PostgreSQL 与 Neo4j 的数据目录放到独立挂载路径。
- PostgreSQL、Neo4j **不要直接暴露到公网**。
- 优先让 Django 通过本机地址访问：
  - PostgreSQL：`127.0.0.1:5432`
  - Neo4j：`bolt://127.0.0.1:7687`
- 4G 机器上应保守配置 Neo4j 内存，避免面板、数据库与图数据库同时抢内存。

### 4. 部署后端代码与 Python 环境

```bash
cd /srv/wisdom-edu/backend
python3.12 -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt
cp .env.example .env
python manage.py migrate
python manage.py collectstatic --noinput
python manage.py check
```

### 5. 配置 `backend/.env`

后端运行配置统一来自 `backend/.env`。首次上线至少应填写：

- Django 基础：
  - `SECRET_KEY`
  - `DEBUG=False`
  - `ALLOWED_HOSTS=edu.qintsg.xyz,106.14.209.7,127.0.0.1,localhost`
  - `USE_X_FORWARDED_HOST=True`
  - `CSRF_TRUSTED_ORIGINS=https://edu.qintsg.xyz,http://edu.qintsg.xyz`
- PostgreSQL：
  - `DB_NAME`
  - `DB_USER`
  - `DB_PASSWORD`
  - `DB_HOST=127.0.0.1`
  - `DB_PORT=5432`
- Neo4j：
  - `NEO4J_BOLT_URL=bolt://127.0.0.1:7687`
  - `NEO4J_USERNAME`
  - `NEO4J_PASSWORD`
- 跨域前端访问：
  - `CORS_ALLOWED_ORIGINS=https://edu.qintsg.xyz,http://edu.qintsg.xyz,http://127.0.0.1:3000,http://localhost:3000`
- AI 功能（如启用）：
  - `LLM_PROVIDER`
  - `LLM_MODEL`
  - 对应的 API Key

如果当前只是稳定上线演示环境，建议优先使用较省资源的默认配置：

- `GRAPHRAG_EMBEDDER_PROVIDER=hash`
- `KT_USE_GPU=false`

### 6. 使用 Daphne 启动 Django ASGI

```bash
cd /srv/wisdom-edu/backend
source .venv/bin/activate
daphne -b 127.0.0.1 -p 8000 wisdom_edu_api.asgi:application
```

### 7. 配置 systemd 守护进程

```ini
[Unit]
Description=WisdomEdu ASGI Service
After=network.target

[Service]
Type=simple
User=www-data
WorkingDirectory=/srv/wisdom-edu/backend
ExecStart=/srv/wisdom-edu/backend/.venv/bin/daphne -b 127.0.0.1 -p 8000 wisdom_edu_api.asgi:application
Restart=always
RestartSec=3
Environment=PYTHONUNBUFFERED=1

[Install]
WantedBy=multi-user.target
```

启用方式：

```bash
sudo systemctl daemon-reload
sudo systemctl enable wisdom-edu
sudo systemctl start wisdom-edu
sudo systemctl status wisdom-edu
```

### 8. 1Panel 与 frp 的职责划分

在“内网后端 + 106 公网入口 + frps”场景下，推荐职责划分如下：

- `1Panel`：仅负责后端服务器上的 Django / PostgreSQL / Neo4j 运维与本机反代管理。
- `frpc / frps`：负责把内网后端的 `127.0.0.1:8000` 映射到 `106` 服务器的 `127.0.0.1:28000`。
- `106` 服务器 Nginx：负责托管 `frontend/dist`，并将 `/api`、`/media`、`/static`、`/ws` 转发到 `http://127.0.0.1:28000`。

也就是说，当前方案下**不建议**再让后端服务器直接暴露公网 `api` 站点；公网入口统一收敛到 `edu.qintsg.xyz` 即可。

## 106 公网入口服务器部署（Nginx + frontend/dist + frps）

### 1. 构建前端产物

可以在前端服务器本机构建，也可以在本地 / CI 构建后上传 `dist`。本机构建示例：

```bash
cd /srv/wisdom-edu/frontend
npm install
npm run build
```

说明：

- 当前前端生产包默认请求与站点同域的 `/api`、`/media`、`/static`、`/ws`。
- `frontend/vite.config.ts` 里的代理只对开发模式生效，当前默认开发联调目标为 `http://127.0.0.1:8000`；如需临时联调 `frps` / 公网入口，可通过 `VITE_DEV_BACKEND_ORIGIN` 覆盖，如需避开端口占用，可通过 `VITE_DEV_PORT` 覆盖开发端口。
- 若你需要让生产包直接访问其他域名/端口，可在构建前设置 `VITE_BACKEND_ORIGIN` 后重新执行 `npm run build`。
- 对于 `edu.qintsg.xyz` 这类静态部署 + frps 场景，推荐继续保持生产包同域访问，并由 Nginx 统一反向代理后端。

### 2. 前端 Nginx 配置示例

```nginx
server {
        listen 80;
  server_name edu.qintsg.xyz;

        root /srv/wisdom-edu/frontend/dist;
        index index.html;

        location / {
                try_files $uri $uri/ /index.html;
        }

  location /api/ {
    proxy_pass http://127.0.0.1:28000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }

  location /media/ {
    proxy_pass http://127.0.0.1:28000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }

  location /static/ {
    proxy_pass http://127.0.0.1:28000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }

  location /ws/ {
    proxy_pass http://127.0.0.1:28000;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_read_timeout 600s;
  }

  location ~* \.(js|css|png|jpg|jpeg|gif|svg|woff2?)$ {
    expires 7d;
    add_header Cache-Control "public, max-age=604800, immutable";
        }
}
```

如果 `frps` 没有绑定到 `106` 服务器的 `127.0.0.1:28000`，请把上游地址改成你的实际可访问后端入口（例如 `http://edu.qintsg.xyz:28000`）。当前 `dist` 已默认兼容这套“同域静态 + Nginx 反代”的部署方式。

### 3. 前端 HTTPS

```bash
sudo certbot --nginx -d edu.qintsg.xyz
```

## 上线后检查清单

1. 打开首页与 `/login`，确认前端静态资源正常加载。
2. 直接访问 `https://edu.qintsg.xyz/register`，确认刷新后不会再返回 404。
3. 打开 `https://edu.qintsg.xyz/api/docs/`，确认 Swagger 可访问。
4. 登录学生端并访问 `/student/ai-assistant`，确认 `/ws/student/ai/chat` 可建立连接。
5. 在后端服务器执行健康检查：

```bash
cd /srv/wisdom-edu/backend
source .venv/bin/activate
python manage.py check
python tools.py diagnose
```

1. 如果启用了 GraphRAG，确认以下目录可写：

- `backend/runtime_logs/rag/`
- `backend/runtime_logs/rag/qdrant/`

1. 在 1Panel / frp / Nginx 中确认：

- `edu.qintsg.xyz` 证书有效
- `frps` 的 `28000` 上游可达
- `/api`、`/media`、`/static`、`/ws` 反向代理状态正常
- PostgreSQL / Neo4j 容器或服务处于运行中
- 备份策略已启用

## 生产环境建议

- `DEBUG=False`
- 为 `SECRET_KEY` 使用高强度随机值
- `ALLOWED_HOSTS` 只保留真实域名与必要地址
- 大模型密钥只写入 `backend/.env`，不要提交到仓库
- PostgreSQL、Neo4j、Daphne 不直接对公网开放
- 后端服务器优先通过 `1Panel` 做数据库备份与证书续期
- 定期备份以下内容：
  - PostgreSQL 数据
  - Neo4j 数据
  - `backend/media/`
  - `backend/runtime_logs/`
  - `backend/models/MEFKT/`
  - `backend/models/DKT/`
